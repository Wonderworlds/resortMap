## Challenges & Lessons

### 1. AC-13 silently expanded scope in Story 6-4

The "zero `style={{`" acceptance criterion swept across the entire `packages/builder-react/src/` tree, but `ScaleDialog.tsx` and `MapCanvas.tsx` were not listed as tasks in Story 6-4. They required non-trivial migration:
- `ScaleDialog.tsx`: full migration to MUI `Dialog` / `ToggleButtonGroup`
- `MapCanvas.tsx`: SVG `style={{}}` attributes replaced with SVG presentation attributes (`pointerEvents="none"`, `cursor="pointer"`, etc.) — a non-obvious technique the spec didn't mention

**Lesson:** When an AC says "all files in X must Y", enumerate every specific file the criterion touches in the task list. "All" ACs must be scoped to what was actually changed in the story.

---

### 2. `poi.icon` schema change is a functional gap

The icon picker stores the POI icon as a registry key string (e.g. `"restaurant"`) in `poi.icon`. This is a **breaking change** from the previous behavior (URL string). The canvas currently renders a plain circle for all POIs regardless of icon value — the picker works in the panel but the icon is invisible on the canvas.

This was documented as a TODO in Story 6-4's design notes, but it means the feature is visually incomplete from the user's perspective: they pick an icon and nothing changes on the map.

**Action required:** Implement canvas icon rendering in a follow-up story.

## Significant Discoveries

### `poi.icon` is now a registry key — schema breaking change

The `icon` field in the `.gwmap` schema changed semantics from "URL string" to "poi-icons registry key string" (e.g. `"restaurant"`). Any previously authored `.gwmap` file using a URL in `icon` will now be misinterpreted. The MapCanvas renderer does not yet consume this field — it renders all POIs as circles.

**Impact on future work:** The next story touching the canvas should add icon rendering. The schema documentation should be updated before any external consumers parse `.gwmap` files.
